# AI UI/UX Resource Evaluation Playbook

Use this playbook when deciding whether outside UI/UX resources, component libraries, design systems, visual reference sites, or design-oriented agent skills should influence Recipe Vault.

## Core Principle

Begin with the product problem, not the resource.

Ask: what design or development problem does Recipe Vault actually need help solving?

## Phase 1: Understand This App

Record:

1. Product purpose
2. Primary users
3. Critical workflows
4. Platform and implementation stack
5. Current maturity and existing identity
6. Current UI/UX weaknesses
7. Accessibility and technical constraints
8. Success criteria

## Phase 2: Classify The Need

Rank the need before evaluating resources:

- Product-flow research
- Visual-direction research
- Component behavior
- Ready-made implementation
- Design-system structure
- Agent efficiency
- Quality assurance

## Phase 3: Resource Roles

- Builder.io Skills: agent workflow and development-process ideas.
- UI UX Pro Max: generated visual direction and design vocabulary.
- 21st.dev: web component inspiration only; not SwiftUI implementation authority.
- UX Components: component anatomy, state coverage, accessibility, and pattern comparison.
- Refero: real-product flow and hierarchy research.

## Phase 4: Score Fit

Score 0-3 for:

- Product fit
- Platform fit
- Stack fit
- Design fit
- Accessibility
- State coverage
- Engineering value
- Maintenance cost
- Overlap
- Evidence quality

## Phase 5: Choose Adoption Level

- `Adopt`: formally integrate because recurring value is strong.
- `Adapt`: extract concepts into Recipe Vault guidance.
- `Reference`: use on demand.
- `Skip`: avoid because value is low or overlap is high.

For Recipe Vault, prefer Adapt/Reference over installing broad generic UI systems.

## Phase 6: Design Research Workflow

1. Define the user problem and screen or flow boundary.
2. Inspect the current implementation.
3. Select only relevant resources.
4. Gather three to five strong references if needed.
5. Extract principles, not screens.
6. Write a compact design brief.
7. Implement natively in SwiftUI.
8. Validate real UI states and refine.

## Guardrails

- Do not copy screenshots, branding, or proprietary compositions.
- Do not let web resources dictate native iOS behavior.
- Do not optimize novelty ahead of comprehension.
- Do not install everything.
- Separate visual research from implementation authority.
